# Brainstorming - 创意生成

## 简介

Brainstorming skill用于在进行任何创造性工作之前探索用户意图、需求并设计实现方案。通过自然对话将想法转化为完整的设计和规范。

## 使用场景

- 创建新功能
- 构建组件
- 添加功能
- 修改行为
- 需要设计新特性

## 核心功能

### 理解想法
- 检查当前项目状态
- 逐个问题细化想法
- 优先使用多选题
- 一次只问一个问题
- 关注目的、约束、成功标准

### 探索方案
- 提出2-3种不同方案及其权衡
- 以对话方式展示选项
- 首先展示推荐选项并解释原因

### 展示设计
- 分段展示设计（每段200-300字）
- 每段后询问是否正确
- 涵盖：架构、组件、数据流、错误处理、测试

## 使用方法

### 1. 启动Brainstorming

当需要创建新功能时，Claude会自动触发此skill。

### 2. 回答问题

Claude会逐个提问，请：
- 一次回答一个问题
- 可以选择多选题或开放式回答
- 如有不确定，可以询问

### 3. 审查设计方案

Claude会分段展示设计，每段后询问：
- "到目前为止看起来正确吗？"
- 如有问题，可以返回澄清

### 4. 确认最终设计

审查完整设计后，确认：
- 架构是否合理
- 组件是否完整
- 数据流是否清晰
- 错误处理是否充分

## 设计完成后

### 文档化
- 将验证后的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 提交到git

### 实现准备
- 询问："准备好设置实现了吗？"
- 使用 `using-git-worktrees` 创建独立工作空间
- 使用 `writing-plans` 创建详细实现计划

## 关键原则

- **一次一个问题** - 不要用多个问题压倒用户
- **优先多选题** - 比开放式更容易回答
- **严格遵循YAGNI** - 从所有设计中删除不必要的功能
- **探索替代方案** - 在确定之前总是提出2-3种方法
- **增量验证** - 分段展示设计，验证每一段
- **保持灵活** - 如有不清楚的地方，返回澄清

## 注意事项

- 不要跳过问题，假设知道答案
- 不要一次性展示完整设计
- 不要在用户确认前继续下一步
- 如有技术疑问，可以询问技术细节
- 设计文档应保存在 `docs/plans/` 目录

      C) Google + Facebook + Twitter + GitHub

[200-300字]

[继续...]
